\begin{abstract}
This article develops a mathematical view of static analysis for Forth-like
stack languages. The central object is the symbolic stack effect, a typed and
identity-sensitive abstraction of how a word transforms the data stack. Three
operations organize the calculus: composition of linear effects, branch merging
as greatest lower bound in a must-analysis order, and loop summarization by
idempotent effects. These operations provide a basis for reasoning about
sequencing, control-flow joins, and repeated execution in Forth-like programs.
\end{abstract}

\noindent\textbf{Keywords:} Program analysis; static type checking; symbolic
stack effects; stack languages; Forth.

\section{Introduction}

Forth programs are sequences of stack operations, so much of the reasoning is
shifted onto the stack. Traditional stack comments of the form
\texttt{( before -- after )} help, but they remain informal and often do not
provide a compositional account of type refinement or value identity through
stack shuffling.

One response is to treat stack effects as semantic objects rather than mere
comments. The main questions are how to summarize a linear sequence of words,
how to merge alternative branches, and how to compute a stable summary for
repeated execution.

This article formulates these questions in a stack-effect calculus. A
non-clashing stack effect is a pair of symbolic stacks, with a separate clash
element for failure; sequencing is partial multiplication; branch convergence
is a greatest lower bound in a must-order; and loop reasoning is approximated
by an idempotent effect or a local fixpoint approximation.
The calculus developed here follows the line of work
in~\cite{poial1990,poial1991,poial2002,poial2006} and its executable
implementation in~\cite{poial2008}.

\section{Mathematical Setting}

\paragraph{Types and symbolic values.}

Let $(T, \leq)$ be a finite partial order of abstract types. The order is read
as ``is at least as specific as.'' Throughout this article, $<$ denotes the
strict part of this partial order. For example,
\[
\textit{char} < +n < n < x
\]
\[
\textit{a-addr} < \textit{c-addr} < \textit{addr} < u < x
\]
\[
\textit{flag} < x
\]
where $x$ is the least exact stack-cell type.

A symbolic stack item is a pair $(t, i) \in T \times \mathbb{N}$, written
$t[i]$. The index distinguishes symbolic values that share the same type.

Let $\Sigma = T \times \mathbb{N}$. A symbolic stack is a finite sequence in
$\Sigma^*$, with the top of stack on the right. Thus $a\ b\ c$ means that $c$
is topmost.

\paragraph{Effects.}

A stack effect is an element of
$S = (\Sigma^* \times \Sigma^*) \cup \{\emptyset\}$. A non-clashing effect is
written formally as $e = (L \to R)$, or in Forth style as $( L \; -- \; R )$,
where $L$ is the required suffix of the input stack and $R$ is the produced
suffix of the output stack.

The effect should be read relationally. If the current abstract stack has the
form $S L$, then after the effect it becomes $S R$, provided that the symbols
on the boundary can be unified consistently.

We distinguish two special elements: $1 = (\, \to \,)$, the identity effect,
and $\emptyset$, the clash effect produced by incompatible types or impossible
unifications.

\paragraph{Matching of boundary symbols.}

If two boundary symbols are matched, the rule used here selects the more
specific compatible type. Write $t \perp u$ when the two types are
incompatible. Then
\[
\operatorname{match}(t[i], u[j]) =
\begin{cases}
t[k] & \text{if } t \leq u, \\
u[k] & \text{if } u \leq t, \\
\emptyset & \text{if } t \perp u,
\end{cases}
\]
where $k$ is a fresh index.

Operationally, one substitutes the fresh result for both boundary symbols
throughout the effect sequence. Intuitively, the two local boundary variables
become one shared symbolic value.

This operation refines the type while preserving identity through substitution.

\section{The Three Core Operations}

\paragraph{Composition.}

Sequential execution is represented by partial multiplication: if $s(\alpha)$
denotes the stack effect of a word or phrase $\alpha$, then
$s(\alpha \beta) = s(\alpha) \cdot s(\beta)$ whenever the output boundary of
$s(\alpha)$ is compatible with the input boundary of $s(\beta)$.

Operationally, multiplication is computed from the top of stack inward.
Suppose $e_1 = (L_1 \to R_1\, a)$ and $e_2 = (L_2\, b \to R_2)$, where $a$ is
the top output symbol of $e_1$ and $b$ is the top input symbol of $e_2$. If
$\operatorname{match}(a, b) = \emptyset$, then $e_1 \cdot e_2 = \emptyset$.
Otherwise let
$z = \operatorname{match}(a, b)$. Substitute $z$ for both $a$ and $b$, remove
the matched boundary pair, and continue recursively. If one side runs out of
boundary symbols first, the unmatched prefix is carried through.

Composition can be described as a polycyclic monoid: it is associative when
defined, it has identity $1$, and it admits a clash (null) element
$\emptyset$. In the untyped setting this specializes to the algebraic setting
of~\cite{poial1990}. In the typed symbolic case the multiplication becomes
partial and information-sensitive, but the semigroup intuition remains.

\paragraph{Branch merging.}

For control-flow joins we need an information order. Let $\gamma(e)$ be the set
of concrete stack transformations represented by abstract effect $e$, and
define $e_1 \leq e_2$ iff $\gamma(e_1) \subseteq \gamma(e_2)$.

Thus $e_1 \leq e_2$ means that $e_1$ is at least as informative as $e_2$; it
describes a smaller set of possible behaviors and therefore gives a stronger
guarantee.

In this order, branch convergence in a must-analysis is modeled by greatest
lower bound $\operatorname{glb}(e_1, e_2)$.

The direction matters. In may-analysis, one usually merges by least upper
bound, because one wants all behaviors that may happen. Here the goal is the
strongest effect guaranteed on all paths. Therefore the appropriate operator is
$\operatorname{glb}$, not $\operatorname{lub}$.

For a conditional with branch effects $p$ and $q$, the abstract effect is
$(\textit{flag} \; -- \; ) \cdot \operatorname{glb}(p, q)$, up to the exact
guard convention of the language. If $p$ and $q$ are
incomparable, the merge is undefined and the analysis rejects the
conditional as lacking a common guaranteed stack behavior.

\paragraph{Loop invariants.}

Loops require a stable summary (invariant) for arbitrary repetition. Let $b$ be
the effect of one loop body execution. Algebraically, the desired summary
should satisfy $i = i \cdot i$. A local approximation is
$\operatorname{glb}(b, b \cdot b)$, which keeps only the guarantee common to
one and two iterations. This is not a full Kleene star, but it extracts the
repeated, stable part of an effect.
The idempotent fragment is therefore the part used for loop reasoning because
it captures the stable part of the domain~\cite{poial2006}.

\section{Type-System Perspective}

The calculus also has related approaches in static typing for stack-based
low-level languages. Bytecode verification for the JVM tracks stack height and
types across control-flow joins~\cite{leroy2003}. Compositional type systems
for stack-based low-level languages~\cite{saabas2006} pursue a similar goal
with more conventional type-theoretic machinery. Type inference for stack-based
languages~\cite{stoddart1993} is an earlier attempt.

Typical inequalities in the type hierarchy are
\[
\textit{char} < +n < n < x
\]
\[
\textit{a-addr} < \textit{c-addr} < \textit{addr} < u < x
\]
\[
\textit{flag} < x
\]
These inequalities determine which boundary matches are accepted and which
type refinements are propagated through compositions and merges.

One feature of the calculus developed here is the combination of stack shape,
subtype-aware typing, and identity-sensitive symbolic flow in one object. This
combination allows stack shuffling to be analyzed without abandoning the effect
notation used in Forth practice.

\section{Implementation}

The calculus is implemented here as a native Gforth program, a single-file
evaluator that reads a declared type hierarchy, a vocabulary of word
specifications, and a source program, then computes symbolic stack effects for
linear code, conditionals, loops, literals, parser words, and new definitions.
The implementation accepts both legacy built-in control structures and
declarative syntax/effect descriptions, so the same three core operations,
composition, $\operatorname{glb}$, and idempotent loop approximation, appear
directly in the checker. When a composition fails, the checker reports
line-and-column diagnostics together with the offending line and symbolic stack
annotations.

The following demo profile illustrates the checker. Using
types such as \texttt{X}, \texttt{n}, \texttt{char}, \texttt{flag},
\texttt{c-addr}, and \texttt{a-addr}, together with specifications for
\texttt{DUP}, \texttt{SWAP}, \texttt{@}, \texttt{C@}, \texttt{0=}, and the
basic control words, the checker can analyze the following program text:

\begin{verbatim}
: PEEK2 DUP @ SWAP @ ;
: MAYSWAP IF SWAP FI ;
: NLOOP BEGIN DUP 0= WHILE REPEAT ;
: ULOOP BEGIN DUP 0= UNTIL ;
: KEEPIF IF DUP ELSE DUP FI ;
DP DP C@ 0= KEEPIF
\end{verbatim}

The generated log records the inferred effects of the new definitions:

\begin{verbatim}
PEEK2 ( a-addr[1] --  a-addr[1] a-addr[1] )
MAYSWAP ( X[1] X[1] flag --  X[1] X[1] )
NLOOP ( n[1] --  n[1] )
ULOOP ( n[1] --  n[1] )
KEEPIF ( X[1] flag --  X[1] X[1] )
\end{verbatim}

This example shows linear code, conditionals, and loop skeletons handled in one
symbolic framework.

\section{Conclusion}

Static analysis for stack languages can be presented as an algebra based on
three operations: composition for sequencing, greatest lower bound for branch
convergence, and idempotent closure for loop invariants. Once types and
symbolic identities are added, this algebra can function as a type checker for
linear code, conditionals, and loops in Forth-like languages.
