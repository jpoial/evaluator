\title{Declarative Stack-Effect Checking for Stack Based Languages}
\author{Anonymous Author(s)}
\renewcommand{\shortauthors}{Anonymous Author(s)}

\begin{document}

\begin{abstract}
This paper studies static type checking for a Forth-like language whose type hierarchy,
vocabulary, and control syntax are loaded from external specification files. Instead of
executing a program for concrete values, the checker computes symbolic stack effects. Its
core calculus has three operators: sequential composition, branch merging by a
lower-bound operator, and a lightweight one-step loop summary. Control structures are
also specified declaratively, so parsing and effect computation share one specification
framework. Once parsing has identified the control structure of a body and each basic
token occurrence already has an assigned effect, the checker computes a unique symbolic
summary for every larger source fragment. The body is accepted exactly when that
bottom-up computation encounters no clash. We implemented the checker in Gforth and
independently in Java; both implementations read the same input formats. We present the
calculus, prove an exactness theorem for parsed bodies, and evaluate the approach on 29
checked-in tests, custom profiles, and a 4{,}255-line application.
\end{abstract}

\maketitle

\section{Introduction}
Concatenative languages compose programs by juxtaposition. In stack-based variants such
as Forth, each word transforms an implicit operand stack. A natural static summary is
therefore a \emph{stack effect}: it records which stack cells a word consumes and which
cells it produces. In simple cases, informal comments such as \texttt{( n n -- n )} are
already useful documentation in Forth practice and standardization documents
\cite{rather1996evolution,forth2012standard}. The goal of this paper is to give such
annotations an executable static semantics.

The challenge is that the target language is not fixed. A checker must account for an
externally declared type hierarchy, symbolic equality constraints created by stack
shuffles, and control forms whose syntax and behavior may vary by profile. Existing work
provides these ingredients separately: algebraic composition of stack effects, typed
wildcards for symbolic equality constraints, and must-analysis operators for branches and
loops \cite{poial1994algebraic,stoddart1993type,poial2002wildcards,poial2006typing}. This
paper combines them in one source-level checker with declarative parser and control
specifications.

The checker takes three external inputs. A \texttt{types} file defines a finite partial
subtype relation. A \texttt{specs} file assigns effects and parsing behavior to ordinary
words, literals, defining words, and control roles. A program file is then parsed and
analyzed against those specifications. Rather than executing the program for concrete
results, the checker computes a symbolic stack-effect summary for each parsed fragment and
each user-defined word. The same mechanism also produces source-aware diagnostics when
adjacent summaries cannot be composed. In that sense, the implementation follows the
abstract-interpretation idea of executing programs over a finite abstract domain rather
than over concrete values \cite{cousot1977abstract}.

\paragraph{Running example.}
It helps to keep one small linear fragment in mind:
\begin{lstlisting}
1 2 + DUP
\end{lstlisting}
using the specifications
\begin{lstlisting}
literal integer ( -- n )
+ ( n n -- n )
DUP ( x[1] -- x[1] x[1] )
\end{lstlisting}
Starting from an empty stack, the fragment leaves two equal \texttt{n} values on top, so
the checker infers the overall effect
\[
( -- n[1]\;n[1]).
\]
The later sections explain how the same composition machinery extends from this simple
example to branches and loops.

\paragraph{Key concepts.}
A \emph{stack symbol} is a typed placeholder such as \texttt{n} or \texttt{x[1]}; reusing
an index records that two positions denote the same abstract value. A \emph{raw effect} is
the internal symbolic result before index cleanup. A \emph{compacted effect} is the
shorter summary that the implementation stores and reports after deterministic
renumbering and optional suppression of some implicit pairwise indices. Branching uses a
partial meet on effects, so it keeps only the information guaranteed by every branch. The
loop operator uses the same order, but compares one execution of a body with two
executions instead of running a global fixpoint engine.

The core theorem matches the implementation. Once a source body has been parsed and each
basic token occurrence already has an assigned effect, the checker computes one unique
summary for every larger fragment of that body. It accepts exactly when the induced
acyclic equation system can be evaluated without clash, and on success the stored
summaries are the resulting compacted effects.

Two design choices keep the checker small. Types are loaded from an external subtype
graph, and control behavior is described by \texttt{syntax:}/\texttt{effect:} blocks
rather than hard-coded rules. The mathematical content is therefore concentrated in a
small symbolic domain and a few deterministic operators. This approach builds on earlier
work on algebraic stack effects, typed wildcards, must-analysis for stack-oriented
languages, and compositional type systems for stack-oriented languages
\cite{poial1994algebraic,stoddart1993type,poial2002wildcards,saabas2006compositional,poial2006typing}.

Concretely, the paper makes four contributions.
\begin{enumerate}
\item It presents a symbolic effect calculus in which stack symbols carry both subtype information and identity constraints, so stack shuffles can be reasoned about compositionally.
\item It gives a uniform must-style account of branching and looping in that same effect domain: branches merge by a lower-bound operator, while loops are summarized by a lightweight local stabilization test.
\item It proves an exact compositional theorem: once a source body has been parsed and each undecomposed token occurrence already has an assigned effect, the checker accepts exactly when the induced segment equations are clash-free, and on success returns the unique family of summaries satisfying them.
\item It evaluates the calculus as a retargetable checker: the type hierarchy, vocabulary, parser behavior, and control structures are all loaded from external specifications, user-defined words are inferred during analysis, and the artifact is exercised on 29 checked-in tests, several custom profiles, and a 4{,}255-line application.
\end{enumerate}

The rest of the paper follows the structure of the checker itself. Sections~2--5 define
the symbolic domain and the effect operators used during checking. Section~6 shows how
declarative control specifications turn parsed source fragments into equations over those
operators and states the exactness theorem. The remaining sections discuss inference of
user-defined words, relation to prior work, evaluation, and conclusion.

\section{Type Domain}
\label{sec:type-domain}
This section fixes the symbolic vocabulary used throughout the paper. Types are not
built into the checker. They come from the external specification files, and the later
effect operators consult only the finite subtype relation declared there.

\subsection{Type hierarchy}
Let $\Types$ be a finite set of canonical type names equipped with an acyclic subtype relation $\leq$. We read $\tau_1 \leq \tau_2$ as ``$\tau_1$ is at least as precise as $\tau_2$,'' matching the sample declarations
\begin{lstlisting}
rel a-addr < c-addr
rel c-addr < addr
rel addr < x
rel flag < x
rel char < n
rel n < x
\end{lstlisting}
from the repository. Thus \texttt{a-addr} is more precise than \texttt{c-addr}, which is more precise than \texttt{addr}, and so on.

The implementation also permits aliases declared on the same \texttt{type} line; mathematically we quotient those aliases to a shared canonical representative and focus on the induced relation on canonical types. Operationally, the checker only needs the meet of \emph{comparable} types.

\begin{definition}[Comparable meet]
For $\tau,\upsilon \in \Types$, define
\[
\tau \glb \upsilon =
\begin{cases}
\tau & \text{if } \tau \leq \upsilon,\\
\upsilon & \text{if } \upsilon \leq \tau,\\
\text{undefined} & \text{otherwise.}
\end{cases}
\]
\end{definition}

This operation simply returns the more precise of the two comparable types. Incomparable
pairs have no defined result here and therefore cause a type clash.

\paragraph{Intuition.}
Composition and branch merging only need one local question at a stack boundary: can a
single concrete cell satisfy both type constraints? If so, the comparable meet keeps the
strongest shared type information; otherwise checking fails at that boundary.

\subsection{Stack symbols}
Stack effects need to express not only which types appear on the stack, but also when two positions are known to denote the same abstract value. To do that, the checker uses typed stack symbols. A \emph{stack symbol} is a type optionally annotated with a positive index, written $\tau$ or $\tau[i]$. The type denotes an abstract class of stack cells. The index denotes correlation across positions: if the same indexed symbol appears twice, those occurrences refer to the same abstract value constraint, not merely two values of the same coarse type.

For example, the article profile in the repository declares
\begin{lstlisting}
PLUS ( x[1] x[1] -- x[1] )
\end{lstlisting}
which states that both arguments to \texttt{PLUS} and its result are tied to the same abstract type variable. Likewise,
\begin{lstlisting}
DUP ( x[1] -- x[1] x[1] )
\end{lstlisting}
states that duplication preserves the abstract identity of the top stack element.

Unnamed symbols are treated as implicitly fresh.

\section{Stack Effects}
\label{sec:stack-effects}
With the symbolic vocabulary in place, we can describe the checker's summary language. A stack effect records how a word transforms the visible stack, while indices express the equality constraints that survive through shuffles and composition.

\subsection{Syntax}
Let $\Sigma$ be the set of stack symbols. A stack effect is a pair of finite sequences
\[
\pi = (L \; -- \; R), \qquad L,R \in \Sigma^\ast.
\]
The sequence is written in ordinary Forth order: the rightmost element is the top of the stack. For example,
\[
(x[2]\;x[1]\; -- \; x[2]\;x[1]\;x[2])
\]
is the effect of \texttt{OVER}.

The empty effect
\[
\id = (\, -- \,)
\]
acts as the identity transformation.

\subsection{Raw and compacted effects}
Evaluation first builds \emph{raw effects}. These retain every fresh index introduced
during composition, branch merging, and loop summarization. The implementation does not
report those raw effects directly. After each successful operator application, it runs a
deterministic compaction pass: remaining indices are renumbered, explicit indices are
preserved, and an implicit index is suppressed when it occurs at most twice. The result
is a \emph{compacted effect}.

Raw and compacted effects serve different roles in the paper. Raw effects are the precise
semantic objects used in the relational statements below. Compacted effects are the
shorter summaries that the checker actually stores and reports. Compaction is therefore
not fully semantics-preserving for identity constraints: suppressing an implicit pairwise
index forgets that two positions were correlated. When that happens, the compacted effect
is a conservative weakening of the corresponding raw one.

\subsection{Relational intuition for raw effects}
The branch and loop operators are best understood as must-style summaries over concrete stack behavior. To make that intuition precise, fix an infinite set $\mathcal{A}$ of concrete atoms. A \emph{concrete stack cell} is a pair $(a,\tau)$ with $a \in \mathcal{A}$ and $\tau \in \Types$; we read $\tau$ as the cell's exact type. A \emph{concrete stack} is a finite sequence of such cells.

Let $\operatorname{Ind}(\pi)$ be the set of indices that occur in an effect $\pi$. A valuation $\eta$ for $\pi$ is a map from $\operatorname{Ind}(\pi)$ to concrete stack cells. If $u$ is a stack symbol with abstract type $\tau_u$, write
\[
c \models_\eta u
\]
when $c=(a,\tau_c)$ satisfies $\tau_c \leq \tau_u$, and if $u$ carries an index $i$, then additionally $\eta(i)=c$. For a symbol sequence $S=u_1\cdots u_n$ and a concrete stack $s=c_1\cdots c_n$ of the same length, write
\[
s \models_\eta S
\]
iff $c_j \models_\eta u_j$ for every position $j$.

For an abstract effect $\pi=(L -- R)$, define its concretization by
\[
\gamma(\pi)=\{(s,t) \mid \text{there exists a valuation } \eta
\text{ such that } s \models_\eta L \text{ and } t \models_\eta R\}.
\]
Thus $\gamma(\pi)$ is the relation on concrete input/output stacks described by $\pi$. Repeated indices induce equality constraints because the same valuation $\eta$ is used on both sides. Unnamed symbols contribute only type constraints, so renaming fresh indices leaves $\gamma$ unchanged.

Define
\[
\pi_1 \preceq \pi_2
\quad\text{iff}\quad
\gamma(\pi_1) \subseteq \gamma(\pi_2).
\]
Lower elements in this order are more informative: they rule out more concrete behaviors and therefore provide stronger guarantees. This is the order-theoretic viewpoint of abstract interpretation specialized to symbolic stack transformations \cite{cousot1977abstract,poial2006typing}.

\paragraph{Notation guide.}
We read $\gamma(\pi)$ as the concrete meaning of $\pi$, $\pi_1 \preceq \pi_2$ as ``$\pi_1$
is at least as informative as $\pi_2$,'' and $\glb$ as ``keep only the information both
sides guarantee.'' These readings are the intuition behind the branch and loop operators
defined later.

Under this reading, sequential composition propagates guaranteed information through a linear segment, while branch merging seeks information guaranteed by every alternative rather than the union of effects that may arise on some path. Because the checker compacts successful summaries before storing them, the reported effect may be a conservative weakening of the corresponding raw effect in this order.

\section{Sequential Composition}
Juxtaposition is the basic program former in Forth, so sequential composition is the
basic operator of the checker. Intuitively, the checker walks across the touching
boundary between two adjacent effects from top to bottom. At each matched position, it
asks whether one concrete stack cell could satisfy both local constraints and, if so,
keeps the more informative shared symbol.

\begin{definition}[Boundary composition]
Let
\[
\pi_1=(L_1 -- R_1), \qquad \pi_2=(L_2 -- R_2).
\]
Their composition $\pi_1 \compose \pi_2$ is computed from the touching boundary between the output stack $R_1$ and the input stack $L_2$:
\begin{enumerate}
\item If $R_1$ is empty, the result is $(L_2L_1 -- R_2)$.
\item If $L_2$ is empty, the result is $(L_1 -- R_1R_2)$.
\item Otherwise let $a$ be the top element of $R_1$ and $b$ the top element of $L_2$.
  If $a.\mathsf{type} \glb b.\mathsf{type}$ is undefined, composition fails.
  If it is defined, replace both symbols everywhere by a fresh symbol whose type is that meet, remove the matched boundary pair, and continue recursively.
\end{enumerate}
\end{definition}

\paragraph{Intuition.}
The fresh symbol keeps the more specific of the two type constraints. Since lower
elements in the order are more precise, the meet is the more informative symbol. This is
why an output proved as \texttt{a-addr} can still satisfy a later input that only
requires \texttt{c-addr}: the joined boundary keeps the stronger fact \texttt{a-addr}.

Write relational composition of concrete stack transformations as
\[
R_1 ; R_2 = \{(s,u) \mid \exists t.\ (s,t)\in R_1 \text{ and } (t,u)\in R_2\}.
\]

\begin{proposition}[Compositionality of boundary composition]
If $\pi_1 \compose \pi_2$ is defined, then
\[
\gamma(\pi_1 \compose \pi_2) = \gamma(\pi_1) ; \gamma(\pi_2).
\]
\end{proposition}

\begin{proof}[Proof sketch]
The recursive boundary algorithm does exactly the matching needed to eliminate an existentially quantified intermediate stack. At each matched boundary position it replaces the two adjacent symbols by their comparable meet, which is precisely the strongest type constraint satisfied by every concrete intermediate cell that could serve both sides. Shared indices are propagated by the same substitution on both effects, so equality constraints across the boundary are preserved as well. Recursing over the boundary therefore removes the intermediate stack one cell at a time without changing the represented input/output relation.
\end{proof}

This proposition concerns the raw composition operator before compaction. The checker
applies $\compact$ to every successful intermediate result before reusing it.

Returning to the running example, the command-line trace from the repository illustrates
the rule:
\begin{lstlisting}
1 2 + DUP
\end{lstlisting}
with specifications
\begin{lstlisting}
literal integer ( -- n )
+ ( n n -- n )
DUP ( x[1] -- x[1] x[1] )
\end{lstlisting}
The checker prints
\begin{lstlisting}
1   ( --  n )
2   ( --  n )
+   ( n n --  n[1] )
DUP ( n[1] --  n[1] n[1] )
\end{lstlisting}
and concludes that the whole program has effect
\[
( -- n[1]\;n[1]).
\]

\begin{definition}[Multiplication of linear effects]
Let $\Pi_{\mathsf{lin}}$ be the set of compacted linear stack effects and let
\[
\widehat{\Pi}_{\mathsf{lin}} = \Pi_{\mathsf{lin}} \cup \{0\},
\]
where $0$ is a clash element. Define multiplication $\odot$ on $\widehat{\Pi}_{\mathsf{lin}}$ by
\[
0 \odot \pi = \pi \odot 0 = 0,
\]
and for $\pi_1,\pi_2 \in \Pi_{\mathsf{lin}}$,
\[
\pi_1 \odot \pi_2 =
\begin{cases}
\compact(\pi_1 \compose \pi_2) & \text{if } \pi_1 \compose \pi_2 \text{ is defined},\\
0 & \text{otherwise.}
\end{cases}
\]
The identity element is $\id = (\, -- \,)$. This multiplication models concatenation of linear program fragments: if $\alpha$ and $\beta$ are linear sequences of words, then
\[
\sem{\alpha\beta} = \sem{\alpha} \odot \sem{\beta}.
\]
\end{definition}

Viewed this way, $\odot$ is the typed version of the product used in earlier algebraic presentations of stack effects \cite{poial1994algebraic,poial2006typing}. It packages sequencing as a total operation with a clash element, while the checker itself continues to use the underlying partial composition $\compose$ because that formulation matches error reporting and local reasoning more directly. The totalized view is useful mainly as a compact algebraic summary of the linear fragment; the metatheoretic results used later rely only on the operational definition of composition above.

\section{Branching as Meet on Effects}
Sequential composition alone is not enough for control flow. A conditional must summarize several alternatives with one effect that is safe no matter which branch executes.

For branching, the checker therefore uses a partial greatest-lower-bound operation on
whole effects. In lattice terminology, this is a partial meet on compatible effects. We
also call it \emph{must-style}: it keeps only the information that every branch
guarantees. Intuitively, the checker merges branch summaries pointwise after accounting
for stack cells that are preserved unchanged across both input and output. This is
necessary because different branches may manipulate different depths of the stack while
still agreeing on a common surrounding context.

More concretely, if one effect has $k$ more inputs than another, it must also have $k$ more outputs. The implementation then extracts a preserved prefix of length $k$ from the larger effect by requiring each corresponding input/output pair in that prefix to be comparable. After that prefix has been merged, the remaining active slices are unified pointwise. If any comparison fails, the branch summaries are incompatible. On raw effects, this construction yields a common lower-bound-style summary whenever it succeeds. The implementation then applies $\compact$ to the merged effect before storing or reporting it, so the reported branch summary may be weaker than the raw meet when an implicit pairwise correlation is suppressed.

Accordingly, the checker uses a meet rather than a join: it searches for information
common to all paths of a conditional, not the union of behaviors that may arise on some
path \cite{poial2006typing}. This is the branching counterpart of the relational reading
from Section~\ref{sec:stack-effects}.

This operator is used by the \texttt{either} constructor in declarative control semantics. The example
\begin{lstlisting}
: MYCHOICE IF ! ELSE C! FI ;
\end{lstlisting}
is inferred as
\begin{lstlisting}
MYCHOICE ( x a-addr flag -- )
\end{lstlisting}
because the two branches have effects
\[
(x\;a\text{-}addr -- )
\qquad\text{and}\qquad
(x\;c\text{-}addr -- ),
\]
and
\[
a\text{-}addr \glb c\text{-}addr = a\text{-}addr.
\]
The merged effect therefore preserves the more specific address requirement.

\section{Iteration via One-Step Stabilization}
Iteration is the point where a full abstract interpreter would usually introduce a
fixpoint computation. The current checker deliberately stops short of that. It does not
build a global control-flow graph or run a general fixpoint engine
\cite{cousot1977abstract}. Instead, the \texttt{repeat} constructor uses a local
repeated-effect operator that compares one execution of a body with two executions. This
is the exact loop-related contract implemented by the checker.

\begin{definition}[One-step repeated summary]
For an effect $\pi$, define
\[
\pi^\star = \pi \glb (\pi \compose \pi),
\]
when both the composition and the greatest lower bound are defined.
\end{definition}

\paragraph{Intuition.}
Operationally, this is a one-step stabilization test. The effect $\pi$ summarizes one
execution of the repeated body, while $\pi \compose \pi$ summarizes two executions. If
those two summaries admit a common lower bound, the checker keeps that lower bound as the
information guaranteed by both one round and one further round of the body. If no such
lower bound exists, the checker declines to assign a repeated-summary effect.

At the raw level, $\pi^\star$ is the common lower-bound-style summary of one and two executions whenever it is defined. The implementation then applies $\compact$ to the result. In control specifications, the \texttt{repeat} constructor uses exactly this one-step criterion. Its intended reading is not full Kleene closure and does not, by itself, claim correctness for all finite iteration counts. Rather, \texttt{repeat} accepts exactly when the one-step and two-step summaries admit a common guaranteed effect.

The bundled program
\begin{lstlisting}
: MYLOOP BEGIN SWAP OVER WHILE INVERT REPEAT ;
\end{lstlisting}
is inferred as
\begin{lstlisting}
MYLOOP ( flag[1] flag[1] -- flag[1] flag )
\end{lstlisting}
This is the compacted summary currently reported by the artifact for the common information obtained from one and two passes of the repeated body.

\section{Declarative Specifications}
\label{sec:decl-specs}
The previous sections defined the effect domain and its operators. We now connect those operators back to source syntax. The key design choice is that parsing behavior and control structure are mostly specification data, not hard-coded cases in the checker core.

\subsection{Ordinary words, literals, and parser words}
The \texttt{specs} file associates stack effects with words and adds the parsing metadata needed to interpret source text. Representative examples are:
\begin{lstlisting}
OVER ( x[2] x[1] -- x[2] x[1] x[2] )
@ ( a-addr -- x )
CONSTANT parse word define ( x -- )
VARIABLE parse word define ( -- a-addr )
literal integer ( -- n )
\end{lstlisting}

Additional clauses refine the operational role of a word.
\begin{itemize}
\item \texttt{parse word} means the word consumes the next token from the source.
\item \texttt{parse until DELIM} means the word consumes source text until a delimiter.
\item \texttt{define}, \texttt{define colon}, \texttt{define constant}, and \texttt{define variable} mark defining words.
\item \texttt{control ROLE} assigns a control-flow role such as \texttt{if}, \texttt{else}, \texttt{fi}, \texttt{begin}, or \texttt{while}.
\item \texttt{immediate}, \texttt{state interpret}, and \texttt{state compile} constrain where the word is legal.
\end{itemize}

Thus the checker is not only typing tokens; it is also modeling the source-level behavior needed to parse Forth-like syntax in a way that aligns with standard Forth parsing disciplines \cite{forth2012standard}.

\subsection{Control structures}
Control structures are specified with paired \texttt{syntax:} and \texttt{effect:} blocks. This separates two concerns cleanly: the \texttt{syntax:} clause says how a source fragment is segmented, and the \texttt{effect:} clause says how the resulting pieces are summarized. This gives a declarative account of source-level control words, a topic that has also been studied in formal models of Forth control structures \cite{power2004formal}. For example, the sample repository contains:
\begin{lstlisting}
syntax: IF <then branch> [ELSE <else branch>] FI
  effect:
    IF
    either <then branch> <else branch>
\end{lstlisting}

and
\begin{lstlisting}
syntax: BEGIN <loop prefix> WHILE <loop body> REPEAT
  effect:
    repeat <loop prefix> WHILE
    repeat <loop body>
\end{lstlisting}

The \texttt{syntax:} clause describes a control-language pattern using concrete boundary words and metavariables for internal segments. Optional parts are written with square brackets. The \texttt{effect:} block is then compiled into an expression language with the grammar
\[
E ::= \eps \mid \langle s \rangle \mid c \mid E\,E \mid \textsf{either}(E,E) \mid \textsf{repeat}(E),
\]
where $\langle s \rangle$ denotes a named program segment and $c$ denotes the runtime effect of a control word such as \texttt{IF}, \texttt{WHILE}, or \texttt{DO}.

The denotational semantics is:
\begin{align*}
\sem{\eps} &= \id,\\
\sem{\langle s \rangle} &= \text{the inferred effect of segment } s,\\
\sem{c} &= \text{the declared runtime effect of control role } c,\\
\sem{E_1 E_2} &= \sem{E_1} \compose \sem{E_2},\\
\sem{\textsf{either}(E_1,E_2)} &= \sem{E_1} \glb \sem{E_2},\\
\sem{\textsf{repeat}(E)} &= \sem{E}^\star.
\end{align*}

This arrangement separates parsing from typing. Once a source fragment has been segmented according to a \texttt{syntax:} declaration, its meaning is calculated entirely by the effect algebra above. That is the bridge from source-level syntax to the compositional theorem in the next subsection.

\subsection{Compositional characterization}
\label{sec:constraint-view}
We can now state the main theorem in the same granularity as the implementation: one
already parsed body in which each basic token occurrence already has an assigned effect.
Fix a finite parsed source tree $T$ for one source body under a finite
\texttt{types}/\texttt{specs} environment $\Gamma$. The tree $T$ is obtained after
parsing has already recognized sequential fragments and control structures. Its internal
nodes are compound source fragments, and its leaves are token occurrences that the
checker does not decompose further. Assume that every leaf in $T$ already has a known
summary effect in $\Gamma$, for example from a word specification, a literal class, or a
previously inferred definition.

Let $D$ be the set of compacted non-clashing effects together with a distinguished clash
value $0$. For each parsed segment $A$, let $X_A \in D$ be its unknown summary. For a
control template $E$ and an assignment $X$ of summary values to segments,
$\widehat{\sem{E}}_X$ denotes evaluation of $E$ under $X$. We write $X^\ast$ for the
unique solution of the segment equations introduced below.

Write $\overline{\compose}$, $\overline{\glb}$, and $\overline{\star}$ for the totalized implemented operators derived from raw composition, raw branch merge, and raw one-step stabilization: whenever the corresponding raw partial operator is defined, the checker returns its compacted result; otherwise the totalized operator returns $0$, and any occurrence of $0$ in an argument propagates to $0$. These are the operators that actually appear during checking. For a control template $E$, define its totalized denotation by
\begin{align*}
\widehat{\sem{\eps}}_X &= \id,\\
\widehat{\sem{\langle s \rangle}}_X &= X_s,\\
\widehat{\sem{c}}_X &= \text{the declared runtime effect of control role } c,\\
\widehat{\sem{E_1 E_2}}_X &= \overline{\compose}(\widehat{\sem{E_1}}_X,\widehat{\sem{E_2}}_X),\\
\widehat{\sem{\textsf{either}(E_1,E_2)}}_X &=
\overline{\glb}(\widehat{\sem{E_1}}_X,\widehat{\sem{E_2}}_X),\\
\widehat{\sem{\textsf{repeat}(E)}}_X &= \overline{\star}(\widehat{\sem{E}}_X).
\end{align*}

Segment equations are then generated bottom-up from $T$.
\begin{enumerate}
\item A leaf with known effect $\pi$ contributes $X_A = \pi$.
\item A sequential node with children $B$ and $C$ contributes
\[
X_A = \overline{\compose}(X_B,X_C).
\]
\item A control node whose \texttt{effect:} template is $E$ contributes
\[
X_A = \widehat{\sem{E}}_X,
\]
where each segment reference $\langle B \rangle$ in $E$ is interpreted as the corresponding child unknown $X_B$.
\end{enumerate}
Let $S(T,\Gamma)$ denote the resulting finite system of segment equations. Because every right-hand side refers only to proper descendants of $A$, the dependency graph of $S(T,\Gamma)$ is acyclic. The theorem below states that bottom-up execution of the core body evaluator computes exactly the unique solution of this system.

\paragraph{Intuition.}
The theorem says that the checker is not doing anything mysterious beyond the operators
already defined. Parsing produces a tree of source fragments. The equations above attach
one deterministic summary equation to each node of that tree. Because every node depends
only on smaller nodes, the whole system can be solved bottom-up, and that bottom-up
solution is exactly what the implementation returns.

\begin{theorem}[Exactness of segment equations]
\label{thm:segment-equations}
Assume that the parsed source tree $T$ is finite and that every leaf of $T$ already has a fixed summary effect under $\Gamma$. Then:
\begin{enumerate}
\item The acyclic system $S(T,\Gamma)$ has a unique solution $X^\ast$ in compacted effects plus $0$.
\item For every parsed segment $A$, the checker returns a clash exactly when $X^\ast_A = 0$; otherwise it returns the compacted effect $X^\ast_A$.
\item In particular, the checker accepts the whole parsed body if and only if the root summary satisfies $X^\ast_{\mathsf{root}} \neq 0$.
\end{enumerate}
\end{theorem}

\begin{proof}[Proof sketch]
Let
\[
D = \{\text{compacted non-clashing effects}\} \cup \{0\}.
\]
Because every equation in $S(T,\Gamma)$ refers only to proper descendants of a segment,
the dependency graph is acyclic. We therefore reason by induction on the height of a
segment in the parsed tree.

If $A$ is a leaf, its equation is $X_A=\pi$ for the fixed known effect supplied by
$\Gamma$. So its value is uniquely determined, and the checker returns exactly that same
value by lookup.

For the inductive step, assume every proper descendant of $A$ already has a unique value
in $D$ and that the checker returns exactly that value, with clash represented by $0$.
There are two cases.

If $A$ is a sequential node with children $B$ and $C$, then its equation is
\[
X_A=\overline{\compose}(X_B,X_C).
\]
By the induction hypothesis, the child values are uniquely fixed as $d_B$ and $d_C$.
Since $\overline{\compose}$ is deterministic, $d_A=\overline{\compose}(d_B,d_C)$ is the
unique solution for $A$. The checker performs that same totalized sequencing step after
recursively evaluating $B$ and $C$, so it returns clash exactly when $d_A=0$, and
otherwise returns $d_A$.

If $A$ is a control node with template $E$, then its equation is
\[
X_A=\widehat{\sem{E}}_X,
\]
where segment references in $E$ are interpreted using the already fixed descendant values.
The value of $\widehat{\sem{E}}_X$ is uniquely determined by structural recursion on $E$:
the base forms $\eps$, segment references, and control-role atoms are fixed directly, and
the constructors for sequencing, \texttt{either}, and \texttt{repeat} use the
deterministic totalized operators $\overline{\compose}$, $\overline{\glb}$, and
$\overline{\star}$. The checker evaluates the same template by the same recursion, so it
again returns clash exactly when the resulting value is $0$, and otherwise returns that
value.

Thus every segment has a unique value in $D$, and those values form the unique solution
$X^\ast$ of $S(T,\Gamma)$. The exact correspondence with the implementation is the
induction statement itself. Clause~(3) then follows immediately at the root of the parsed
tree.
\end{proof}

The older inequality-based presentation of stack effects in
\cite{poial1991algebraic,poial1992multiple} can still be recovered as a
relaxation of the raw, uncompacted operators:
replace each equation $X_A = F_A(\vec{X})$ by an inequality
$X_A \preceq F_A(\vec{X})$ \cite{poial1991algebraic,poial1992multiple}. For the
present checker, however, the exact theorem concerns the compacted implemented
summaries and is the acyclic equation characterization above rather than a
least-fixpoint result over inequalities.

\paragraph{Scope of the theorem.}
Theorem~\ref{thm:segment-equations} is intentionally local. It characterizes one body
after parsing has already built its tree of source fragments and after every token
occurrence that the checker treats as basic already has an assigned effect. The full top-level checker does more than that local
semantics alone: it grows the specification dictionary as definitions are encountered, may
seed provisional summaries for forward references, and supports a dedicated
\texttt{RECURSE} path during definition checking. Those implementation conveniences are
used to make whole-source checking practical, but they are not part of the statement of
Theorem~\ref{thm:segment-equations}. The theorem covers the body-evaluation phase that
those outer mechanisms repeatedly invoke.

\section{Inference of User-Defined Words}
The local theorem above is enough to explain how the checker grows its dictionary during a whole-file pass. At top level, defining words are interpreted immediately, and the body semantics of Theorem~\ref{thm:segment-equations} is used to assign a summary to each newly introduced word. A colon definition starts a new sequence, parses until its terminating control word, computes the effect of the body, and installs the resulting effect as the specification of the new word. Constants and variables are handled similarly but with special defining shapes:
\[
\texttt{CONSTANT} : (x --), \qquad \texttt{VARIABLE} : (-- a\text{-}addr).
\]

The implementation therefore supports the standard Forth workflow in which the specification dictionary grows during analysis \cite{forth2012standard}. Inferred definitions are written to a log file; for the article profile in \texttt{testdata/positive/article/}, the current artifact records:
\begin{lstlisting}
MYLOOP   ( flag[1] flag[1] -- flag[1] flag )
MYCHOICE ( x a-addr flag -- )
MYTEST   ( a-addr[2] x[1] flag -- a-addr[2] x[1] a-addr[2] )
MYFI     ( x[1] x[1] flag -- x[1] x[1] )
MYAGAIN  ( a-addr -- x )
MYSEQ    ( a-addr[2] x[1] -- x[1] a-addr[2] x )
MYSEQ2   ( a-addr[1] a-addr[1] -- )
\end{lstlisting}

These examples demonstrate several features of the checker at once.
\begin{itemize}
\item \texttt{MYCHOICE} uses branch merging over comparable address types.
\item \texttt{MYTEST} shows prefix-aware branch merging when the alternatives manipulate different active depths of the stack.
\item \texttt{MYLOOP} and \texttt{MYAGAIN} illustrate the current one-step loop summary together with the implementation's compaction of implicit indices.
\item \texttt{MYSEQ} and \texttt{MYSEQ2} show how sequential composition propagates typed correlations through larger definitions, including the polymorphic helper word \texttt{PLUS}.
\end{itemize}

These examples also show the practical effect of compaction: some implicit pairwise correlations are intentionally not retained in the stored summaries. The point is not that every inferred summary is maximally precise, but that the same small collection of operators is sufficient to infer useful summaries across ordinary sequencing, control flow, and user-defined words.

\section{Relation to Prior Work}
This paper sits at the intersection of several lines of work on stack-oriented typing. Early algebraic treatments of Forth stack effects studied sequential composition, multiple stack effects, and solvability via systems of inequalities \cite{poial1991algebraic,poial1992multiple,poial1994algebraic}. Typed wildcards and subtype-sensitive matching then provided a way to track both abstract type precision and symbolic identity across stack shuffles \cite{stoddart1993type,poial2002wildcards}. Later work described branch merging and loop summaries in must-analysis terms, using greatest lower bounds and repeated-effect stabilization criteria \cite{poial2006typing}. These papers supply most of the individual ingredients used here; the present contribution is to combine them into one source-level checker with a precise theorem for what that checker computes.

difference is the combination of declarative control specifications, parser behavior,
user-defined-word inference, and an exact theorem for parsed-body checking in one
checker. In particular, relative to \cite{saabas2006compositional}, the present work
stays in the source-language idiom of Forth \cite{forth2012standard}, where control
syntax itself is part of the specification interface.

WebAssembly is a natural comparison point. Modern Wasm work includes official validation
rules, mechanised metatheory, and refined accounts of stack polymorphism
\cite{webassembly2026spec,watt2018wasm,mcdermott2022wasm}. The setting is different:
WebAssembly validates a fixed bytecode language with built-in structured control
operators, whereas the present checker analyzes source text whose parser words, defining
words, and control forms are supplied by external specifications. The emphasis here is
therefore on declarative \texttt{syntax:}/\texttt{effect:} descriptions and source-aware
diagnostics rather than bytecode validation alone.

Compared with \cite{power2004formal}, the focus here is static source checking rather
than a full semantic model of control-word execution.

\section{Evaluation}
We evaluate four claims: the checker behaves correctly on the checked-in tests,
produces useful diagnostics, retargets across declarative profiles, and scales beyond
toy examples. The repository contains both the Gforth implementation discussed throughout
the paper and an independent Java implementation that consumes the same
\texttt{types}/\texttt{specs}/\texttt{prog} formats. All quantitative results below come
from batch-running the checked-in tests through the Java implementation.

\paragraph{Tests.}
The repository contains 29 tests. Thirteen are expected to type-check successfully and
16 are expected to produce diagnostics. The positive set consists of 7 tests under the
bundled Forth 2012 profile, 5 under custom type/specification profiles, and 1 article
example used in this paper. Table~\ref{tab:corpus-results} summarizes the results.

\paragraph{Results.}
The Java implementation accepts all 13 positive tests and rejects 14 of the 16 negative
tests. The two remaining misses are \texttt{linear\_clash.fs} and
\texttt{top\_level\_clash.fs}, both linear numeric/address clashes involving
\texttt{DP 1+}. These cases expose gaps in top-level linear reasoning rather than in the
declarative control machinery.

\paragraph{Diagnostics.}
The negative tests cover malformed control nesting (\texttt{missing\_fi},
\texttt{missing\_repeat}, \texttt{unexpected\_else}), unknown words, branch-summary
incompatibility, loop-body incompatibility, nested-definition rejection, unterminated
definitions, and multiple-error recovery. Diagnostics include file positions and the
offending source span. For example, on \texttt{if\_clash.fs} the checker reports that the
two \texttt{IF} alternatives have effects \texttt{( -- a-addr )} and
\texttt{( -- char )} and highlights the exact \texttt{IF ... ELSE ... FI} region.

\paragraph{Retargetability.}
The custom-profile tests exercise features that are not hard-coded into the checker core: custom literal handling, custom type hierarchies, aliasing profiles, and declaratively specified control structures. In particular, the bundled \texttt{WHEN ... OTHERWISE ... ENDWHEN} example and the switch-like example in \texttt{positive/custom\_types/} are described through \texttt{syntax:}/\texttt{effect:} declarations rather than evaluator-specific control code. Together with the literal-alias, literal-count, and literal-scanner examples, these tests show that the same effect calculus can be reused across profiles with substantially different vocabularies and surface syntax.

\paragraph{Scale.}
The main scale indicators are straightforward. The bundled
Forth 2012 environment consists of a 50-line type file and a 278-line specification file,
for 328 lines of declarative typing and control information in total. Under that
environment, the checker analyzes the single-file implementation
\texttt{gforth-evaluator.fs}, which has 4{,}255 lines. The artifact also contains an
independent Java implementation totaling 12{,}138 lines across its checked-in
\texttt{.java} sources. These figures are enough to show that the checker is being applied
to a nontrivial specification environment and a multi-thousand-line program rather than
only to toy fragments.

\paragraph{Limitations and scope.}
The checker is a static evaluator, not a full runtime. It relies on the declared
specification set, rejects nested defining words inside definitions, enforces
interpretation-state restrictions, treats the external type hierarchy as authoritative,
and uses the one-step repeated-summary criterion of Section~5 rather than a general loop
fixpoint. The tests therefore demonstrate coverage of the intended source-language
fragment, not industrial benchmarking. Within that scope, they show both the range of
supported examples and two remaining precision gaps.

\section{Conclusion}
The current formulation isolates a small symbolic core for typed stack-effect
checking in a Forth-like setting. The case studies suggest that declarative
control descriptions and symbolic effects are expressive enough to support
source-level checking across several profiles while still producing useful
diagnostics.

Several directions remain open. The local loop-summary operator of
Section~5 could be replaced by stronger invariant generation or abstract-interpretation
techniques. The present theorem is also deliberately local to one parsed body with
known leaf effects; extending it to recursive definitions, forward references, and
top-level dictionary growth would give a more complete account of the implementation.
More broadly, it would be useful to mechanize the metatheory and to compare the
resulting framework more systematically with other typed stack languages, e.g.
WebAssembly.
